I Built an AI Product Owner Agent That Reviews My Azure DevOps Backlog

Chris Phelan  ·  Senior Technical Product Manager

Every product leader knows the feeling. You open your backlog and it's a graveyard of half-written user stories, bugs with no description, and acceptance criteria that says "it works." You know you should clean it up. You also know you won't, because you have 40 other things to do today.

So I built an agent to do it for me.

What it does

I set up Claude CLI as a product owner agent that connects to Azure DevOps, pulls every open user story, bug, feature, and epic, and reviews each one against a defined set of completeness criteria. It generates a markdown report that shows me exactly what's wrong with each item, suggests corrected descriptions in "As a... I want/should/need... So that..." format, and writes real Given/When/Then acceptance criteria derived from the context of each story.

Items with serious problems get tagged NeedsReview in DevOps so they're visible to the team. Items with minor gaps (missing estimates, title formatting) show up in the report but don't get tagged.

On the next run, it checks everything it's tracking for changes, flags new issues, and removes tags from items that have been fixed. It keeps a local state file so it knows what changed since last time.

How it works

The setup is surprisingly simple. Three things:

  • A project folder with a CLAUDE.md file that contains the agent's instructions — what to check, how strict to be, what format to use for suggestions, and how to interact with Azure DevOps.
  • The Azure CLI with the azure-devops extension, authenticated and pointed at your org and project.
  • Claude CLI, launched from the project folder. It reads the CLAUDE.md automatically and knows what to do. You tell it "run the initial import" and it starts pulling items via az boards query, fetching details, and building the report.

That's it. No custom code. No infrastructure. No API integration to maintain. The entire "application" is a markdown file with well-structured instructions.

What the agent checks

The criteria are split into two tiers so it's not flagging everything on day one.

Flags (tags the item in DevOps): missing descriptions, descriptions not in user story format, missing acceptance criteria, acceptance criteria not in Gherkin format, missing severity on bugs, missing parent links on stories.

Warnings (report only): title format inconsistencies, missing hour estimates, missing category tags, feature/epic gaps. These are aspirational for now and get promoted to flags as the team matures.

The tiers matter. My first run without them tagged 98% of the backlog. That's not useful — it's noise. Tuning the strictness so that flags represent genuinely broken items and warnings represent improvement opportunities made the output actionable.

What the suggestions look like

The agent doesn't just say "description is wrong." It rewrites it. If a user story says "Add filtering to alerts page," the agent suggests something like:

As a Shop Supervisor, I want to filter alerts by severity level, so that I can prioritize critical vehicle issues during my daily review.

Then it generates acceptance criteria based on that description:

Given I am a Shop Supervisor viewing the alerts dashboard When I apply a filter for severity "Critical" Then only alerts with severity score 80-100 are displayed Given I am a Shop Supervisor viewing the alerts dashboard When I clear all filters Then all active alerts for my location are displayed

The suggestions use real domain terminology because the CLAUDE.md includes context about the product, its personas, and its capabilities. That's the difference between generic AI output and something a PM can actually approve and push.

What this saves

The obvious answer is time. A manual backlog grooming pass across 100+ items takes hours. The agent does it in minutes and produces a structured report I can review over coffee.

But the real value is consistency. Every item gets the same level of scrutiny every time. No one skips the boring bug that's been sitting there for three sprints. No one forgets to check if acceptance criteria are testable. The bar doesn't drop when the team is busy.

It also creates a feedback loop. When developers see that vague stories get flagged automatically, they start writing better stories. The agent doesn't replace the PM — it raises the floor so the PM can focus on strategy instead of formatting.

What I'd do differently

Start with loose criteria and tighten over time. I started too strict and the output was overwhelming. Flag the things that genuinely block work (no description, no AC) and warn on everything else until the team builds the habit.

Include your product context in the CLAUDE.md. Without it, the agent writes generic suggestions. With it, the suggestions reference your actual personas, features, and domain language. That's what makes them worth approving instead of rewriting.

Scope it to one team first. I started with our web team's area path. Get the criteria right for one team before rolling it out to the org.

What's next

Right now I run it manually when I want a backlog health check. The next step is scheduling it to run daily via Claude CLI's headless mode, either through a pipeline or a scheduled task, so the report lands before standup. The longer-term goal is triggering it on work item creation so new items get reviewed immediately.

If you're a PM or PO drowning in backlog debt, this is a weekend project that pays for itself immediately. The hardest part isn't the technology — it's deciding what "good" looks like for your team and writing it down clearly enough for the agent to enforce it.
